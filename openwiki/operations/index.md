# Operations

This section covers local development, deployment, database migrations, environment configuration, and troubleshooting.

---

## Local Development

### Prerequisites

- **Node.js** 22.x (verified locally; slightly older/newer may work)
- **Supabase CLI** (optional; needed only for database migrations)
- **A Supabase project** (recommended; app also runs read-only on seed data without one)

### Setup

1. **Clone the repository**:
   ```bash
   git clone https://github.com/shakrunk/cinemarchive.git
   cd cinemarchive
   ```

2. **Install dependencies**:
   ```bash
   npm install
   ```

3. **Create `.env.local`** with your Supabase project credentials:
   ```
   VITE_SUPABASE_URL=https://<project-ref>.supabase.co
   VITE_SUPABASE_ANON_KEY=<your-anon-key>
   ```

   - Get these from your Supabase project: Settings → API → Project URL & anon key
   - TMDB/OMDb API keys are **not** here — they live on the Edge Function as secrets
   - No `.env.local` needed for local-only (seed data) mode

4. **Start the dev server**:
   ```bash
   npm run dev
   ```

   - Opens at `http://localhost:5173`
   - Hot module reloading (HMR) enabled
   - Changes to source files instantly reflected in browser

### Available Commands

```bash
npm run dev        # Start dev server (HMR)
npm run build      # Type-check + production build → dist/
npm run preview    # Preview production build locally
npm run lint       # Run ESLint (configured in .eslintrc.js)
```

### Running Without Supabase

If you don't have a Supabase project or `.env.local`:

- App loads **seed data** from `/src/store/mockData.ts`
- All features work except persistence (no saves to DB, no auth)
- Useful for prototyping / local design work

## Database Migrations

### Philosophy

Schema changes are **version-controlled migrations**, never direct SQL editor edits.

### Workflow

1. **Create a migration**:
   ```bash
   supabase migration new add_some_feature
   ```
   - Creates timestamped file: `supabase/migrations/20260707000000_add_some_feature.sql`

2. **Edit the migration**:
   ```sql
   -- supabase/migrations/20260707000000_add_some_feature.sql
   CREATE TABLE new_table (
     id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     ...
   );
   ALTER TABLE new_table ENABLE ROW LEVEL SECURITY;
   -- ... RLS policies ...
   ```

3. **Test locally** (Docker required):
   ```bash
   supabase db push
   ```
   - Applies migration to local Postgres
   - Tests the migration before pushing

4. **Push to main branch**:
   ```bash
   git add supabase/migrations/...
   git commit -m "feat(db): ..."
   git push origin main
   ```

5. **CI/CD runs** (`.github/workflows/db-migrate.yml`):
   - Supabase CLI runs `db push` on production
   - Migration applied automatically

6. **Update `schema.sql`**:
   - Keep `/schema.sql` in sync with the current state
   - It's a human-readable reference, not used by deployments
   - Can be regenerated with `supabase db pull` (Docker required)

### Migration Best Practices

- **One migration per logical change** — makes rollback easier
- **Timestamp naming** — auto-generated by CLI, ensures order
- **Test RLS policies** — migrations define them
- **Avoid DDL in migrations that depend on FK** — order matters
- **Never edit old migrations** — create new ones instead
- **Include comments** — explain why a change is needed

## Supabase Configuration

### Project Setup

1. Create project at https://supabase.com
2. In Supabase dashboard:
   - **Settings → API**: Copy `Project URL` and `anon key` → `.env.local`
   - **Settings → Edge Functions**: Deploy `media-proxy` and `redeem-invite` (see below)
   - **Settings → Authentication**: Enable passkey/WebAuthn provider
   - **Auth → Email Templates**: Customize signup/invite emails (optional)

### Edge Functions

#### Deploy media-proxy

```bash
supabase functions deploy media-proxy
```

- Sets secrets (environment):
  ```bash
  supabase secrets set TMDB_API_KEY=<key>
  supabase secrets set OMDB_API_KEY=<key>
  ```

- Proxy all TMDB/OMDb requests through this function
- Caches responses in `api_cache` table (24-hour TTL)

#### Deploy redeem-invite

```bash
supabase functions deploy redeem-invite
```

- Server-side invite validation
- Creates users on valid code + email
- No secrets needed (uses service role key built-in)

**Note**: Function changes deploy independently of migrations. Editing `index.ts` without deploying leaves the live function stale.

### Row Level Security

All tables have RLS enabled (see `/schema.sql`). Key policies:

- **Own data**: Authenticated users see/edit only their own rows
- **Shared links**: Valid `shared_access_keys` token holders (with optional `share_scopes` filters)
- **Friends**: Users connected via `friendships.state = 'friend'` can see each other's libraries

Every query is automatically filtered. A hacked client can't bypass RLS — it's enforced at the DB layer.

Required repository secrets for CI/CD:

- `SUPABASE_ACCESS_TOKEN` — For `supabase db push` and function deploys
- `GITHUB_TOKEN` — Auto-injected for GitHub Pages deploy

## Authentication & Passkeys

### Passkey/WebAuthn Flow

1. **Signup**:
   - User clicks "Sign up"
   - Enters email, clicks "Send invite code"
   - Code sent via email (external email service assumed)
   - User enters code → calls `redeem-invite` Edge Function
   - Server creates user, sends notification to inviter
   - User redirected to passkey registration
   - Browser prompts for passkey (fingerprint, PIN, etc.)
   - Passkey stored in browser/device

2. **Login**:
   - User clicks "Sign in"
   - Enters email
   - Browser prompts for passkey
   - Supabase verifies passkey challenge
   - User authenticated

3. **Logout**:
   - User clicks "Sign out"
   - Clears Supabase session + Zustand user state

### Invites

- **Invite codes** stored in `invite_codes` table
- Each owner can generate codes with optional redemption limit (or uncapped)
- Codes are 8-char uppercase hex (derived from UUID)
- Redeem function tracks attempts (rate limiting) and redeemed count

Source: `/src/lib/auth.ts`, `/supabase/functions/redeem-invite/index.ts`

```env
VITE_SUPABASE_URL=https://<project>.supabase.co
VITE_SUPABASE_ANON_KEY=<anon-key>
```

- `VITE_` prefix: available to frontend (Vite exposes them)
- Never commit secrets (API keys, service role key, etc.)

Set via Supabase CLI or dashboard:

```bash

supabase secrets set OMDB_API_KEY=<key>
```

- Not available to frontend (server-side only)
- Accessed via `Deno.env.get('TMDB_API_KEY')` in functions

## Performance Tuning

### Client-Side

- **Virtualization**: Poster grid doesn't virtualize (assumes < 10K titles)
- **Memoization**: `filteredTitles` memoized, stat computation cached
- **Code splitting**: Views/modals lazy-loaded by Vite
- **Image optimization**: Posters hosted by TMDB (not bundled)

### Database

- **Indexes**: Thoughtfully selected for common queries (see `/schema.sql`)
- **RLS policies**: Tested to ensure they use indexed columns
- **Denormalization**: Activity feed denormalized for performance (triggers populate)
- **Caching**: API responses cached 24 hours (Edge Function)

### Network

- **Edge Functions**: TMDB/OMDb calls proxied through Edge Function with caching
- **Shared links**: Read-only access via RLS (no additional auth layer)
- **Pagination**: Activity feed paginated (50 items per page)

## Troubleshooting

### Build Fails: "Cannot find module X"

- Run `npm install` to ensure all dependencies are installed
- Check that `.env.local` exists with `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` (or remove if testing without Supabase)

### Type Errors: "ts(2304) Cannot find name 'X'"

- Run `npm run build` (or `tsc -b`) to check full type context
- Some type errors only appear in the full build, not in IDE

### Network Error: "CORS Origin Not Allowed"

- Edge Functions have CORS headers (`Access-Control-Allow-Origin: *`)
- Check Supabase project is linked in `.env.local`

### "Cannot read property 'user' of undefined"

- Auth not initialized yet. App waits for `authChecked` state before rendering main view.
- Check Supabase credentials in `.env.local`

### Notification Not Appearing

- App polls `fetchUnreadNotificationCount()` every few seconds
- Notification inserted by DB trigger, may take a moment to appear
- Refresh page to force a poll

### RLS Policy Rejection: "new row violates row level security policy"

- Check the policy definition in `/schema.sql`
- Ensure `user_id` matches authenticated user's ID
- Log the exact query + row data to debug

## Maintenance

### Regular Tasks

1. **Monitor Supabase usage**: Check dashboard for DB size, function invocations
2. **Review migrations**: Ensure pending migrations are tested before pushing
3. **Update dependencies**: `npm outdated`, then `npm update` cautiously (test after)
4. **Check for errors**: Monitor GitHub Actions logs for failed deploys

### Database Backups

- Supabase auto-backs up daily (Pro plan)
- Use `supabase db pull` to export schema + data locally (requires Docker)

### Testing Edge Functions Locally

Edge Functions can be tested locally before deploying:

```bash
supabase start
supabase functions serve media-proxy
```

- Then call `http://localhost:54321/functions/v1/media-proxy` with test data

## Advanced: Running Migrations Without Docker

If `supabase db push` fails due to Docker requirement:

```bash
supabase migration list  # See all migrations, which are applied
supabase db push --remote-only  # Push directly to production (risky without local test)
```

**Caution**: Only use `--remote-only` if you're confident in the migration. Always test locally first if possible.
